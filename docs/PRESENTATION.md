## Слайд 7 — Математическое обеспечение

**Содержание слайда:**

**Заголовок:** Основные виды математического обеспечения

**10 блоков (2 ряда по 5):**

1. **Генерация эмбеддингов изображений** — f_img: ℝ^{H×W×C} → ℝ^d (CLIP ViT)
2. **Генерация эмбеддингов текста** — f_text: tokens → ℝ^d (текстовый энкодер CLIP)
3. **L2-нормализация** — v̂ = v / ‖v‖₂ (единичные векторы)
4. **Косинусное сходство** — sim(q̂, v̂ᵢ) = q̂ · v̂ᵢ ∈ [−1, 1]
5. **Пороговая фильтрация** — Results = {i | sim ≥ τ}
6. **HNSW-граф** — логарифмическая сложность поиска O(log n)
7. **Сегментная адресация** — offset(k) = k × d × sizeof(float32)
8. **CRC32 checksum** — целостность сегментных файлов
9. **Adaptive efSearch** — динамическая регулировка параметров поиска
10. **Device profiling** — автоматический выбор Lite/Standard/Pro модели

# С4 контейнерная диаграмма системы

Архитектура описывает мобильное приложение для семантического поиска по медиатеке, работающее локально на устройстве.

Пользователь взаимодействует с мобильным приложением, разработанным на Kotlin Multiplatform. Приложение отображает интерфейс, реализует бизнес-логику и координирует обработку данных.

Для работы с фотографиями приложение обращается к системной медиатеке устройства — PhotoKit на iOS или MediaStore на Android. Через эти API оно читает медиафайлы и их метаданные.

Далее изображения обрабатываются локально с помощью встроенного ML-движка ОС: CoreML на iOS или NNAPI на Android. ML-модель, например через TensorFlow Lite Delegate, строит векторные представления изображений, которые затем используются для семантического поиска.

Метаданные файлов, альбомы и теги сохраняются в локальной базе данных SQLite. Сами векторные индексы хранятся отдельно в хранилище индексов в виде бинарных HNSW-файлов, что позволяет быстро искать похожие изображения по смыслу.

Для защиты данных приложение использует системное хранилище ключей — Keychain на iOS или Keystore на Android. Оно применяется для шифрования локальных данных и индексов.

Ключевая особенность архитектуры — вся обработка выполняется на устройстве, без отправки фотографий в облако. Это обеспечивает конфиденциальность, офлайн-работу и быстрый семантический поиск по медиатеке.

### Слайд 6 — Технологический стек

Основной язык — Kotlin. Для UI используется Jetpack Compose с Material 3. Для базы данных — Room поверх SQLite. Для ML-инференса — TensorFlow Lite с поддержкой GPU и NNAPI делегирования. Для векторного поиска реализован алгоритм HNSW на чистом Kotlin. Фоновая индексация управляется через WorkManager с учётом заряда батареи и температуры устройства.

### Слайд 7 — Математическое обеспечение

Математическое обеспечение системы основано на проекции изображений и текстовых запросов в единое многомерное пространство признаков с помощью модели CLIP.

Изображение преобразуется в вектор размерностью до 512 через визуальный энкодер. Текстовый запрос — через текстовый энкодер той же модели. Оба вектора L2-нормализуются, после чего вычисляется косинусное сходство — скалярное произведение нормированных векторов.

Для эффективного поиска используется алгоритм HNSW с логарифмической сложностью, обеспечивающий время отклика менее 50 миллисекунд даже на базе из 50 тысяч фотографий.

### Слайд 8 — Модель данных

Локальная база данных содержит шесть сущностей. Центральная — MediaItemEntity, хранящая метаданные файлов и статус индексации. IndexingJobEntity обеспечивает очередь задач эмбеддинга. VectorSegmentEntity и VectorPostingEntity связывают метаданные с бинарными сегментами.

Ключевое проектное решение — векторные дескрипторы вынесены за пределы SQLite в отдельные бинарные файлы, к которым осуществляется mmap-доступ через MappedByteBuffer. Это позволяет эффективно работать с медиатеками большого объёма без нагрузки на оперативную память.

### Слайд 9 — Реализованный функционал

Приложение реализует шесть основных модулей функциональности.

Семантический поиск позволяет вводить запросы на естественном языке и получать ранжированные результаты. Фоновая индексация автоматически сканирует галерею и генерирует эмбеддинги с контролем ресурсов. Сегментный движок обеспечивает mmap-чтение и атомарную публикацию сегментов. Адаптивный ML автоматически выбирает подходящую модель в зависимости от устройства. Timeline отображает медиатеку с группировкой по датам. Экран настроек позволяет управлять исключёнными директориями и переиндексацией.

### Слайд 11 — Тестирование

Для контроля качества написано более 68 JVM unit-тестов, покрывающих модули базы данных, векторного поиска, машинного обучения и конкурентного доступа.

Отдельный benchmark-suite измеряет латентность поиска, пропускную способность индексации, recall и потребление памяти. Recall@10 алгоритма HNSW относительно полного перебора составляет не менее 0,95, что подтверждает высокое качество приближённого поиска.

### Слайд 12 — Развёртывание

Схема развёртывания включает единственный физический узел — мобильное устройство пользователя. Внутри него развёрнуты три подсистемы: клиентское приложение с Compose-интерфейсом, локальный ML-движок с TFLite и NNAPI, и локальное хранилище данных — SQLite через Room плюс бинарные сегменты с mmap-доступом.

Серверные компоненты полностью отсутствуют. Вся обработка — от генерации эмбеддингов до поиска ближайших соседей — выполняется исключительно на устройстве.
